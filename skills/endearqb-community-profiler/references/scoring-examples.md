# 评分校准案例

## 内容质量评分典型案例

### 满分案例（内容质量 90-100）

**发言示例：**
> "这个内存泄漏是因为 Go runtime 的 finalizer 触发时机不确定导致的。可以用 `runtime.SetFinalizer` + `runtime.GC()` 强制触发，但更推荐改用 `sync.Pool` 来管控生命周期。贴一个能复现的最小 demo：[代码块]"

**评分解析：**
- 技术深度：25/25（定位到 runtime 层，给出机制解释）
- 信息密度：23/25（提供了可操作方案 + 代码）
- 原创性：25/25（个人分析 + 解决思路）
- 表达清晰度：24/25（逻辑清晰，可直接复用）

---

### 中等案例（内容质量 40-60）

**发言示例：**
> "可以试试用 Redis 做缓存，这样能快很多"

**评分解析：**
- 技术深度：12/25（方向对但无原理）
- 信息密度：10/25（建议模糊，缺细节）
- 原创性：15/25（常见建议，非针对性分析）
- 表达清晰度：18/25（表达清楚但内容薄）

---

### 低分案例（内容质量 < 20）

**发言示例：**
> "哈哈哈确实" / "[表情包]" / "我也遇到过这个问题"

**评分解析：**
- 技术深度：0/25
- 信息密度：2/25
- 原创性：5/25
- 表达清晰度：10/25（表达无问题但内容为空）

## 专业权威信号识别案例

### 正信号示例

✅ **解答了他人问题：**
> A问："为什么 Dockerfile 每次都全量重建？"
> B答："你的 COPY . . 放在 npm install 之前了，改一下层顺序，把 package.json 先 COPY 进来"

✅ **纠正了错误信息：**
> C说："用 `== null` 判断就行"
> D说："注意 TypeScript strict 模式下 `== null` 会同时 match undefined，如果只想判断 null 要用 `=== null`"

✅ **给出可验证的结论：**
> "刚测了一下，在 Node 20 上 `structuredClone` 比 `JSON.parse(JSON.stringify())` 快约 40%，数组大时差距更明显"

### 负信号（不给专业权威加分）

❌ "大佬们帮看看这个报错"（提问，不是解答）
❌ "转一篇好文" + 链接（传播，非权威）
❌ "我感觉应该是 xxx 的问题"（猜测，无依据）

## 角色分类边界案例

### 问题解决者 vs 核心贡献者

**问题解决者**：专注回答问题，但很少主动发起讨论、分享观点。
- 互动中回复他人占比 > 60%
- 综合分可能 < 70（因为活跃度和社区粘性中等）

**核心贡献者**：既解答问题，也主动发起高质量讨论。
- 综合分 ≥ 70
- 内容质量和专业权威双高

### 潜水者 vs 新人

**潜水者**：长期存在于群内，只是不说话（入群时间长，发言 < 5）
**新人**：最近才加入，发言数少是因为入群时间短

如无法判断入群时间，统一标注"低活跃"，不武断分类。

---

## 数据不足时的处理原则

- 发言 < 3 条：所有维度标注"样本不足"，仍给出估计值但置信度低
- 内容只有链接/图片：内容质量维度无法有效评估，标注"内容类型受限"
- 无时间戳信息：活跃时段分析跳过，社区粘性维度降权计算
